01What it does

Babatunde Awoyemi runs two GitHub identities — a personal profile and the Techbase Consultant Services organisation — and this workflow keeps every repository across both current: a real modernization pass on structure and design, a distinct colour palette per repo, and a rewritten, professional README. A repo directory file tracks what exists, where it lives, and its rebuild status, so nothing gets rebuilt twice or skipped.

02The linking architecture

This is the rule every rebuild protects. Every individual project repo — every course, tool or app that isn't the portfolio or a profile itself — links to exactly three places and nothing else:

1 · Personal portfolio

babatundeawo.github.io — the single site that lists every project, personal and Techbase, with a live link and a repo link each.

2 · Personal profile README

github.com/babatundeawo — kept in sync with the portfolio's full project list.

3 · Techbase org profile

github.com/techbaseng — the only place that links onward to the real company website, techbasengr.com.ng.

No repo cross-links to a sibling project. Land on any one project and you can only trace back to who made it and which org it belongs to — the full catalogue only exists in the two portfolio locations.

03Rebuild session flow

  • Confirm ownership. Which profile the repo belongs to, its current status, and whether it's a redirect stub — checked against the repo directory before anything else.
  • Propose an identity. A distinct colour palette and typography direction for that repo, unless one is already established.
  • Rebuild. Modernize the structure and design, rewrite the README with the three-link footer, and strip every other outbound link.
  • Summarize and flag. What changed, plus anything needing a decision — a new repo type, unclear ownership, and so on.
  • Remind to update the record. The repo directory's status column, and the portfolio's project list if it's a new addition.

04Standing style rules

  • Clean, modern type stacks — never default system fonts.
  • Light-theme-first, with a distinct accent colour per project.
  • Copy that reads naturally, never assembled — no em or en dashes in written content.
  • One repo, or one closely related batch, per session — never a dozen at once in a single conversation.
